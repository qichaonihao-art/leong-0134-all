我现在要在现有的“声音克隆工具网站”基础上继续升级功能。请你先完整理解下面的产品目标，再基于这个目标去设计和实现，不要只做表面修改。

当前项目原本已经先接通了智谱的声音克隆流程，是为了优先跑通上传音频、克隆音色、输入文案、生成语音、试听和下载这条闭环。

现在我要把这个工具升级成一个“支持多家声音克隆模型接口”的版本，接入三家能力：
1. 智谱
2. 阿里云
3. 火山引擎

请你围绕这个目标来改造现有工具。

---

## 一、模型接入与选择

我希望在开始克隆前，用户可以手动选择当前要使用哪一家声音克隆模型接口。

也就是说，页面上应该有一个明确的模型选择入口，例如：
- 智谱
- 阿里云
- 火山引擎

用户在新建音色时，先手动选择当前要使用哪一家，然后上传样本音频，再进行克隆。

---

## 二、三个独立 API 输入区

因为要接入三家接口，所以页面不能再只有一个 API Key 输入框，而是要支持三家各自独立的 API 配置。

我希望页面上能有三个独立的 API Key 输入入口，分别对应：
- 智谱 API Key
- 阿里云 API Key
- 火山引擎 API Key

请继续沿用当前“保存在浏览器本地、刷新后还能继续使用”的思路。
也就是说，每个用户只在自己的浏览器里保存自己的三个 API Key，不需要多人共享，不需要服务端保存。

---

## 三、历史音色必须绑定来源模型

这是非常关键的逻辑：

以后每个已保存的音色，不能只是一个普通名字或样本记录，还必须保存它的“来源模型信息”。

例如：
- 某个音色是通过智谱克隆出来的
- 某个音色是通过阿里云克隆出来的
- 某个音色是通过火山引擎克隆出来的

这样做的原因是：
以后用户再次选中某个历史音色时，系统必须自动继续使用它原本所属的模型接口。

也就是说：
- 历史音色如果来自智谱，后续继续生成文案语音时，就自动走智谱
- 历史音色如果来自阿里云，就自动走阿里云
- 历史音色如果来自火山引擎，就自动走火山引擎

不要让历史音色和模型来源脱钩。

请把“音色来源模型”作为音色数据结构里的必要字段保存下来。

---

## 四、模型选择与历史音色之间的逻辑优先级

这里请你把交互逻辑设计清楚：

1. 当用户是“新建音色”时：
   - 由当前页面手动选择的模型决定这次走哪一家接口

2. 当用户是“选择历史音色继续生成语音”时：
   - 优先使用这个历史音色原本绑定的来源模型
   - 不要因为顶部当前模型切换选项变化，就错误地改用另一家接口

简单来说：
“新建音色看当前所选模型，历史音色看它自己的来源模型”。

---

## 五、生成结果需要标明来源模型

因为以后会混合使用三家接口，所以最终生成出来的音频结果，必须能清晰区分是由哪一家生成的。

请在生成结果区域中，给每个生成的语音结果增加清晰的来源标识，例如：
- 智谱
- 阿里云
- 火山引擎

这个标识应该在试听和下载区域附近明显展示，让用户一眼就能知道这段音频来自哪家模型。

如果最终下载的文件名也能带上模型来源标识，会更好，但请优先保证界面上能明确区分。

---

## 六、页面结构优化：我的音色改为右侧折叠面板

当前主界面项目太多，显得比较杂乱。

我希望把“我的音色”这个区域从当前主界面中移开，改成放到右侧的折叠面板里。

设计目标是：
- 平时主界面保持简洁
- 需要时用户自己手动展开右侧折叠面板
- 在折叠面板里选择历史音色即可

---

## 七、折叠音色面板中要按模型来源分组

因为以后会有三家的历史音色，所以折叠面板里的“我的音色”不能再混在一起显示。

请按照模型来源进行分组展示，例如：
- 智谱音色
- 阿里云音色
- 火山引擎音色

这样用户在查看和选择历史音色时，一眼就知道这个音色属于哪一家。

请确保分组展示清晰、层级合理、便于选择，不要让信息混乱。

---

## 八、继续保持当前工具的已有优点

在改造过程中，请不要破坏当前已经做好的这些体验：

- API Key 本地保存
- 文案输入框高度自动根据内容调整
- 生成完成后不自动清空文案
- 已克隆音色可连续复用
- 音色本地保存
- 生成语音后可试听和下载
- 页面整体以“内部工具、简单易用”为目标

---

## 九、这次改造的目标不是做复杂平台，而是做成更清晰的多模型工具

请你理解，这次不是要把项目做成复杂的大型平台，而是：
在现有已跑通的基础上，把它升级成一个更合理、更清晰、更可持续使用的“多模型声音克隆工具”。

重点是：
1. 三家接口接入
2. 三个 API Key 输入
3. 音色必须绑定来源模型
4. 历史音色继续使用原模型
5. 生成结果明确标注来源模型
6. 我的音色移动到右侧折叠面板
7. 折叠面板里按三家模型分组展示音色

请先根据这些目标，梳理清楚产品结构、数据结构和交互流程，再开始实现。
不要只做表层 UI 改动，要确保逻辑完整、可执行、可使用。

另外，请你先不要一下子重构得太激进，优先基于当前代码结构完成这次多模型扩展，并确保原有功能不被破坏。
